  Two separate systems, no bridge:                                                                                                                                                               
                                                                                                                                                                                                 
  1. UI/BDD tests — .feature files can resolve ${login:email} / ${registered:email} / ${default:email} from tests/config/test-config.json:48-63 (testCredentials.default,                        
  testCredentials.registered). Documented at mcp-server/docs/TEST_MCP.md:598-648.                                                                                                                
  2. API tests (the ones pre-push runs, served by http://localhost:8766) — read only tests/api_definitions/*.json. They support three placeholder systems:                                       
    - {{dynamic:email}} — random per run                                                                                                                                                         
    - {{cache:…}} — value captured from a prior test's response (captureResponse)                                                                                                                
    - Hard-coded literals like "password": "TestPass@1234"
                                                                                                                                                                                                 
  What they don't support: ${default:email} / ${registered:email} / ${login:email} inside api_definitions/*.json. The bridge never happens. test-config.json.testCredentials is effectively      
  ignored by the API runner.                                                                                                                                                                     

  OAuth: test-config.json:114-121 has authentication.oauth.tokens.{accessToken,refreshToken,idToken} slots. They're placeholders only — no code reads them, no flow captures them. Manual-paste
  instructions exist in TEST_MCP.md:694-719 but there's no auto-capture / write-back.

  Pre-flight check: none. mcp-server/run-all-tests.sh:81-160 creates a config if missing but never validates testCredentials.default against the running app. The pre-push hook
  (mcp-server/templates/pre-push:630-655) POSTs directly to /api/test/start with zero user readiness checks. Result: login test fails → 9 dependents cascade with "User not found", and the user
  sees a generic red report.

  The gap

  Four missing pieces — in priority order:

  1. API-definition payloads cannot reference test-config.json. Even if creds are there, the runner doesn't resolve ${default:email} in api_definitions/*.json.
  2. No pre-flight "is the test user real?" check. The runner never tries POST /api/auth/login with the configured default before firing the suite.
  3. No self-heal. If the default user doesn't exist in the target DB, nothing registers one, persists the generated credentials back to testCredentials.registered, and continues.
  4. No OAuth bootstrap. bypassEnabled: false and empty tokens mean OAuth-gated APIs silently 401. No "launch browser, capture token, write back" step.
